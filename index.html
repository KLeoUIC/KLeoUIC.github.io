Standard

Concise

Blocking send that completes when the send buffer can be safely reused. Completion may occur before a matching receive is posted (eager protocols) or after the receive begins (rendezvous), and the exact behavior is implementation dependent. To avoid deadlock in tightly coupled exchange patterns, prefer using nonblocking operations (e.g., `MPI_Isend`/`MPI_Irecv`), `MPI_Sendrecv`, or adjust ordering. Messages are matched by communicator, source/destination, and tag; ordering between messages from the same sender to the same receiver on the same communicator is preserved when tags are equal. Counts and datatypes must be type-matched at the receiver. A send to `MPI_PROC_NULL` returns immediately and has no effect. For very large counts, use the large-count variant `MPI_Send_c`.

Performs a blocking send in standard mode. The call may block until the message data has been copied out of the user buffer, which may occur eagerly or only after the matching receive is posted; once it returns, the buffer may be safely reused. Users should avoid circular wait patterns that can deadlock with blocking sends and consider nonblocking operations when overlapping communication with computation.

_MPI_Send_ performs a standard-mode blocking send. The message data is the specified number of datatype entries from the send buffer; a count of zero sends an empty data part. The message envelope uses the sending process, destination rank, tag, and communicator to determine where the message can be received. Tags must be within the valid range for the communicator, and destination ranks are interpreted in that communicator's destination group. Completion of _MPI_Send_ means the application may reuse the send buffer. It does not imply that the matching receive has completed, and a standard send may either complete before a matching receive is posted or block until the receive has begun. Programs should not rely on implementation buffering for correctness; use buffered, synchronous, ready, or nonblocking communication when those semantics are required.

_MPI_Send_ performs a blocking standard-mode send to the destination rank and tag in the communicator. The call returns only after the message data and envelope have been safely stored away so the send buffer can be modified, but completion may occur either because the data was copied to a matching receive or because MPI buffered it internally. Standard-mode send is nonlocal: it may complete before a matching receive is posted, or it may block until a matching receive starts or buffer space becomes available, so portable programs should not depend on internal buffering.

`MPI_Send` is a blocking standard-mode send. It does not return until the send buffer can be safely reused by the caller, meaning the data has either been copied into a system buffer or delivered directly to the receiver's buffer. Whether MPI buffers the outgoing message is entirely implementation-defined. If buffer space is available, the send may complete before the receiver has posted a matching receive. If no buffer space is available, the call blocks until a matching receive is posted and the data has been moved. Programs must not rely on system buffering to avoid deadlock: every correct standard-mode send program must remain correct when all sends are replaced with `MPI_Ssend`. If guaranteed buffering is required, use `MPI_Bsend` with a user-provided buffer attached via `MPI_Buffer_attach`. Because completion may depend on a matching receive being started by the destination, `MPI_Send` is _nonlocal_. Patterns where both sides block in a send waiting for the other to receive can deadlock. Nonblocking sends (`MPI_Isend`) avoid this by allowing the caller to continue, and also enable overlap of communication with computation. Sending a message to `MPI_PROC_NULL` is always valid and is a no-op. Sending a message to a process's own rank on an intra-communicator is permitted but is unsafe with the blocking send: if no matching receive is already posted, the call may block indefinitely.

Performs a blocking send in the standard communication mode; the call does not return until the send buffer is safe to reuse, either because the message was copied into a system buffer or transferred directly into the matching receive buffer. MPI may buffer the outgoing message, allowing the call to complete before a matching receive is posted, or may require a matching receive to be started before completing; portable programs must not rely on buffering to avoid deadlock. Users should use nonblocking operations or alternative send modes to avoid circular-wait deadlocks and to improve overlap of communication with computation.

This is the blocking send operation in standard mode. The send call does not return until the message data and envelope have been safely stored away so that the sender is free to modify the send buffer. In standard mode, it is up to MPI to decide whether outgoing messages will be buffered. MPI may buffer outgoing messages, and in such a case, the send call may complete before a matching receive is invoked. If buffer space is unavailable or MPI chooses not to buffer for performance reasons, the send call will not complete until a matching receive has been started, and the data has been moved to the receiver. A standard mode send can be started whether or not a matching receive has been started, and its successful completion may depend on the occurrence of a matching receive. Users should not rely on system buffering for correctness of standard mode sends, as any system can run out of buffer resources. Correct, portable programs must not depend on system buffering.

This function performs a blocking send in the standard communication mode. It does not return until the message data and envelope have been safely stored away, allowing the sender to safely reuse or modify the send buffer. Users should ensure a matching receive is or will be posted to prevent deadlocks from circular wait patterns.
